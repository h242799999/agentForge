# Spec Reviewer Skill

> 通过 `/spec-reviewer [代码路径]` 触发，对照设计文档检查代码实现是否与规格一致。
> 文档由 `specs/INDEX.md` 管理映射关系，支持大型文档集（> 200 页）的按节精确加载。

---

## 前置条件

项目根目录需存在以下结构（首次使用运行 `/spec-indexer` 自动生成）：

```
specs/
├── INDEX.md          ← 模块 → 文档章节的映射表（核心）
└── {设计文档文件}    ← 本地存放的文档（Markdown / 纯文本）
```

远程文档（Google Drive / Confluence）需先下载到 `specs/` 目录后使用。

---

## 调用语法

```
/spec-reviewer                              # 审查当前目录所有代码
/spec-reviewer src/payment/                 # 审查指定目录
/spec-reviewer PaymentViewModel.kt          # 审查指定文件
/spec-reviewer --doc api-design.md          # 只对照指定文档审查
/spec-reviewer --module payment             # 只审查指定模块
```

---

## 执行步骤

### Step 0：加载通用规则

Read `skills/review-commons/RULES.md`

---

### Step 1：验证 specs/INDEX.md 存在

```bash
ls specs/INDEX.md 2>/dev/null || echo "NOT_FOUND"
```

若不存在，输出提示：
```
未找到 specs/INDEX.md。
请先运行 /spec-indexer 生成文档索引，或手动创建。
参考模板：specs/INDEX.md
```

---

### Step 2：读取 INDEX.md，定位相关文档章节

Read `specs/INDEX.md`

根据审查目标（代码路径 / 模块名）在索引中匹配：
- 精确匹配：代码路径包含索引中的路径关键词
- 模糊匹配：代码文件名 / 类名与索引中的模块名相似

输出匹配到的规格条目，例如：
```
匹配到以下规格：
- payment/ → api-design.md §3 支付接口
- PaymentViewModel → detailed-design/payment.md §2 支付流程状态机
```

---

### Step 3：按节加载规格文档

**对于每个匹配到的规格条目：**

```bash
# 1. 找到章节起始行
grep -n "^## §3\|^## 3\.\|^# 第三章" specs/api-design.md

# 2. 找到下一章节起始行（确定章节范围）
# 3. 使用 Read 工具指定 offset 和 limit 只加载该章节
```

> 只加载相关章节，不加载整个文档，支持大型文档（> 200 页）

---

### Step 4：读取被审查的代码

按 Tier 优先级读取代码文件（参考 commit-reviewer 的分级策略）：
- Tier 1（接口定义、数据模型、核心业务逻辑）：全量读取
- Tier 2（UI、工具类）：读前 100 行 + 关键函数

---

### Step 5：规格符合性审查

对照已加载的规格章节，逐项检查代码实现：

#### 5.1 API 设计符合性

| 检查项 | 具体内容 |
|--------|---------|
| 接口路径 / 方法名 | 代码中的 API 路径是否与 API 设计文档一致 |
| 请求参数 | 参数名、类型、是否必填是否与文档一致 |
| 响应结构 | 返回的数据结构、字段名、类型是否与文档一致 |
| 错误码 | 错误码定义是否与文档中的错误码表一致 |
| 认证方式 | 接口的认证要求是否正确实现 |

#### 5.2 详细设计符合性

| 检查项 | 具体内容 |
|--------|---------|
| 业务流程 | 代码逻辑是否覆盖了详细设计中的所有流程步骤 |
| 状态机 | 状态转换是否与状态机图一致，是否有遗漏的状态 |
| 数据模型 | 实体字段定义、类型、约束是否与数据模型一致 |
| 异常处理 | 文档中定义的异常场景是否都有对应的处理逻辑 |
| 边界条件 | 文档中明确说明的边界条件是否在代码中处理 |

#### 5.3 UI / 様式书符合性

| 检查项 | 具体内容 |
|--------|---------|
| 组件名称 | 代码中的组件与様式书中的组件对应关系是否正确 |
| 交互行为 | 点击、滑动等交互是否与様式书描述一致 |
| 数据绑定 | UI 显示的字段是否与设计文档中的数据来源一致 |
| 错误状态 | 错误页面、空状态是否按文档实现 |

#### 5.4 未实现 / 超出规格

| 检查项 | 具体内容 |
|--------|---------|
| 规格遗漏 | 文档中定义但代码中未实现的功能点 |
| 超出规格 | 代码实现了文档中未定义的行为（需确认是否需要补充文档） |
| 文档过时 | 代码与文档不一致，但代码明显是正确的（文档可能需要更新） |

---

### Step 6：应用通用规则

- 应用 RULES.md「维度 A」对审查代码做代码逻辑检查
- 应用 RULES.md「维度 C」检查代码规范

---

### Step 7：输出报告

```markdown
# 规格符合性审查报告

**审查范围**：`{代码路径}`
**对照文档**：`{文档名 + 章节列表}`
**审查时间**：`{日期}`
```
